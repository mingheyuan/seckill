单机更强，不是“单机算力更高”，而是你现在这套链路里“双机新增的分布式开销”大于“横向扩容收益”。

先看你自己的结果对比（同样 30s，失败率都接近 0）：

单机 1000 VU：15846 req/s，P95 26.91ms
双机 1000 VU：10911 req/s，P95 73.78ms
单机 1500 VU：18249 req/s，P95 62.39ms
双机 1500 VU：11126 req/s，P95 154ms
这已经很明显：双机吞吐平台在 11k 左右，单机还能到 18k。

为什么会这样：

请求路径变长了
双机场景里，proxy 在 131，但 layer 有一部分在 132，很多请求变成“131 -> 132 -> Redis(可能再跨机) -> 132 -> 131”。
单机场景基本是本机回环和本机容器，RTT 更低、抖动更小。

你引入的是“分布式协调成本”，不是纯算力叠加
双机后多了：跨机 TCP、Redis Cluster 路由/节点状态同步、Nacos 实例发现与健康变化。
这些组件在高并发下都会放大尾延迟（P95/P99 先变差）。

双机曾出现过真实链路不稳证据
你日志里出现过 17002 timeout/refused、proxy 长时间 502/超时，这类瞬时故障会让队列堆积，吞吐直接掉台阶。
单机没有这类跨机网络故障面。

入口仍是单点 proxy
即使 layer 扩到双机，入口没有横向扩，很多压力仍先卡在同一个 proxy 的连接池/调度上。
所以双机没有把入口瓶颈一起扩掉。

业务是“轻计算+高频短链路”
秒杀主流程是 Redis/Lua 快操作，这类负载对“网络延迟和稳定性”非常敏感。
当单机还没打满时，跨机开销会直接吃掉本该获得的扩容收益。

一句话总结：
你现在的双机形态更像“分布式成本实验”，还不是“完整横向扩容架构”，所以单机在当前阶段跑得更快、更稳是合理结果。

如何解决“入口仍是单点 proxy”

核心原则：把 proxy 也做横向扩容，并且给 proxy 前面再加一层高可用入口。

推荐拓扑：

Client/k6 -> VIP(Keepalived) -> HAProxy/Nginx(2台) -> proxy(131:8080,132:8080) -> layer/redis/mysql

关键点：

1. proxy 多实例
- 131 和 132 都启动 proxy（你当前脚本已具备这个能力）。
- 确认两台 proxy 都健康：/healthz 返回 200。

2. 前置负载均衡不单点
- 不要让 k6 或客户端只打某一台 proxy IP。
- 使用 2 台 LB（HAProxy 或 Nginx）+ Keepalived 漂移 VIP，避免“LB 自身成为新单点”。

3. LB 做健康检查和连接复用
- 对后端 proxy 做 HTTP 健康检查（/healthz）。
- 开启 keepalive，减少短连接建立成本。
- 失败实例自动摘除，恢复后自动加入。

4. 压测入口改为 VIP
- BASE_URL 改成 VIP 地址，而不是 131 或 132 的单机地址。
- 这样才能验证“入口已横向扩容”的真实收益。

最小可用 HAProxy 示例（LB 节点）：

frontend fe_seckill
	bind *:8080
	mode http
	default_backend be_proxy

backend be_proxy
	mode http
	balance roundrobin
	option httpchk GET /healthz
	http-check expect status 200
	server proxy131 192.168.32.131:8080 check
	server proxy132 192.168.32.132:8080 check

验证标准（建议）：

1. 可用性验证
- 手动停掉一台 proxy，压测不断流，总体吞吐下降但服务不中断。

2. 性能验证
- 同一 VU 下，对比“单 proxy 入口”与“VIP 入口”的吞吐和 P95。
- 若入口瓶颈被解除，应看到吞吐提升或同吞吐下延迟下降。

3. 资源验证
- 观察两台 proxy CPU、连接数是否更均衡，而不是一台打满一台空闲。

补充：

如果只做“layer 双机”，但入口仍单点，系统上限通常仍会被入口卡住；
只有把入口和后端一起横向，双机收益才会稳定体现。

第一主问题是单个 proxy 入口
所有请求先过同一入口，且前置中间件都在这里执行，见 main.go:30 和 main.go:31。这会先形成上限。

第二主问题是 MySQL 侧容量与查询路径
MySQL 连接池上限目前只有 20，见 mysql_redis_store.go:46。高并发下会排队。
用户订单查询按 user_id 查，但表里只看到 activity_id+user_id 复合唯一索引，查询语句见 mysql_redis_store.go:132，这类场景常见需要补 user_id 索引来避免扫描。

还有一个容易被忽略的并发点是 proxy 限流锁
限流是全局 mutex，见 ratelimit.go:33，并发再上去会放大锁竞争。
