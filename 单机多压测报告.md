1.单机压测下，CPU/网卡/内存是共享的
你把 layer 从 1 个加到 2 个，还是在同一台机器上抢同一套资源，通常不会线性提升吞吐，反而会多出进程切换和调度开销。

2.入口还是单 proxy，网关先成为瓶颈
所有流量都先过 proxy，限流和安全校验都在这里做。你之前大量 429 就是这个点导致的。相关逻辑在 ratelimit.go:1。

3.Redis/MySQL 仍是共享后端，扩 layer 不会扩后端能力
不管几个 layer，都在打同一个 Redis 和同一个 MySQL。秒杀里热点 key + Lua + 写流/落库会把压力集中到后端。
相关路径在 mysql_redis_store.go:1。

4.你起了 Redis Cluster，但业务代码并没有真正“按集群客户端方式”使用
当前代码是 redis.NewClient（单节点客户端），不是 NewClusterClient，所以“开了集群容器”不等于业务真实跑在 cluster 模式。
可见 mysql_redis_store.go:33 和 worker.go:35。

5.成功率下降通常来自网关/后端保护，而不是 layer 数量
出现 429/502/超时时，k6 就判失败。你之前失败率高，主要是网关限流和瞬时抖动，不是“多 layer 本身把成功率打坏”。

6.你的双 layer 负载分配是轮询，但收益被上面瓶颈抵消
即使已经轮询（http.go:121），在单机+共享后端场景里仍可能看不到提升。
